# Binary Delivery Spec

## Overview

Gosh consists of three components, each a separate binary:

|Component    |Repository    |Description                                                |
|-------------|--------------|-----------------------------------------------------------|
|`gosh` (CLI) |gosh-ai-cli   |User entry point, manages the other components             |
|`gosh-agent` |gosh-ai-agent |MCP server, task execution                                 |
|`gosh-memory`|gosh-ai-memory|Memory storage, runs in Docker                             |

All binaries are published in GitHub Releases of their respective repositories.

-----

## 1. Platforms

|Target                     |CLI|Agent|Memory    |
|---------------------------|---|-----|----------|
|`x86_64-unknown-linux-gnu` |✅  |✅    |✅ (Docker)|
|`aarch64-unknown-linux-gnu`|✅  |✅    |✅ (Docker)|
|`x86_64-apple-darwin`      |✅  |✅    |✅ (Docker)|
|`aarch64-apple-darwin`     |✅  |✅    |✅ (Docker)|

Windows support is planned — see spec `windows_support`.

Memory runs in Docker on all platforms (Linux, macOS).
Landlock is applied inside the Docker container — the host OS does not matter.

-----

## 2. Release Artifacts

Each release (`vX.Y.Z`) contains:

```
gosh-{version}-{target}.tar.gz          # archive (Linux/macOS)
gosh-{version}-{target}.tar.gz.sha256   # SHA-256 checksum
manifest.json                           # manifest with checksums for all artifacts
```

Example for CLI:

```
gosh-v0.5.0-x86_64-unknown-linux-gnu.tar.gz
gosh-v0.5.0-x86_64-unknown-linux-gnu.tar.gz.sha256
gosh-v0.5.0-aarch64-apple-darwin.tar.gz
gosh-v0.5.0-aarch64-apple-darwin.tar.gz.sha256
manifest.json
```

## 3. Security Model

### GitHub Releases Manifest

**Trust chain:**

```
install.sh
  └── GitHub API: GET /repos/{owner}/{repo}/releases/latest
        └── manifest.json from release assets
              └── sha256 from manifest → verification of downloaded binary
```

**Hardcoded in install.sh:**

```bash
GITHUB_ORG="gosh-sh"
CLI_REPO="gosh-ai-cli"
GITHUB_API="https://api.github.com"
```

**Fetching the manifest:**

```bash
RELEASE=$(curl -fsSL \
  -H "Accept: application/vnd.github+json" \
  "$GITHUB_API/repos/$GITHUB_ORG/$CLI_REPO/releases/latest")
# From the response: tag_name → version, assets → manifest.json URL + binary URL
```

**Each component has its own manifest in its own repo.** The manifest contains checksums for all platforms of the current release:

```json
{
  "version": "0.5.0",
  "format_version": 1,
  "requires": {
    "gosh-agent": ">=0.5.0",
    "gosh-memory": ">=0.4.0"
  },
  "artifacts": {
    "x86_64-unknown-linux-gnu": {
      "sha256": "<hash>",
      "archive": "gosh-v0.5.0-x86_64-unknown-linux-gnu.tar.gz"
    },
    "aarch64-apple-darwin": {
      "sha256": "<hash>",
      "archive": "gosh-v0.5.0-aarch64-apple-darwin.tar.gz"
    }
  }
}
```

The archive URL is not needed in the manifest — it is derived from the release assets (manifest and binaries reside in the same release).

The `requires` field is only present in the CLI manifest — it knows which versions of agent and memory are compatible.

**The manifest is generated by CI** automatically when a GitHub Release is created: CI collects checksums of all artifacts and publishes `manifest.json` as a release asset alongside the binaries.

**Behavior when GitHub API is unavailable:**

```
if GitHub API is unavailable:
    → abort with explicit error:
      "Cannot fetch release manifest: GitHub API unreachable.
       Check your network connection or try again later.
       Manual download: https://github.com/gosh-sh/gosh-ai-cli/releases"
    → DO NOT proceed with installation without verification
```

No silent fallback. No installation without manifest verification.

**Independent verification by the user:**

The manifest and checksums are public in GitHub Releases — anyone can download manifest.json and verify the binary's sha256 manually.

### Future: Blockchain Manifest

> **Status: backlog**

In the future, the manifest may additionally be published to a blockchain (Acki Nacki) for independent verification that does not depend on GitHub. This does not replace the current mechanism but adds an additional layer of trust.

## 4. CLI Installation

### Linux / macOS

```bash
curl -fsSL https://raw.githubusercontent.com/gosh-sh/gosh-ai-cli/main/install.sh | bash
```

### Windows (PowerShell) — planned

> Windows builds are not yet available. See spec `windows_support`.

### What the install scripts do

**install.sh (Linux/macOS):**

```
1. Detect OS and architecture
2. GET GitHub API /repos/.../releases/latest → fetch manifest.json from assets
   - If GitHub API is unavailable → abort with explicit message
3. Extract sha256 and archive name for the current platform from the manifest
4. Download the archive from the same release
5. Verify SHA-256 against the value from the manifest
   - If mismatch → abort with error
6. Extract and install the binary:
   - Linux: sudo install → /usr/local/bin/gosh
   - macOS: sudo install → /usr/local/bin/gosh + xattr -d com.apple.quarantine
7. Print first-run instructions
```

**macOS: Gatekeeper/quarantine.** Downloaded binaries receive the `com.apple.quarantine` xattr.
install.sh runs `xattr -d com.apple.quarantine /usr/local/bin/gosh` automatically.
Long-term — proper code signing via Apple Developer ID.

**install.ps1 (Windows) — planned:** see spec `windows_support`.

## 5. CLI Manages Components

After installation, the CLI manages downloading and updating the agent and memory:

```bash
gosh setup          # idempotent — installs/refreshes agent + memory; for CLI prints curl one-liner
```

### Docker preflight: gosh setup

`gosh setup` checks for Docker presence and readiness before installing components.

**Check algorithm:**

```
1. Run `docker info`
   - If command not found → print platform-specific instructions (see below) → abort
   - If command exists but daemon is not running → "Docker is installed but not running" → abort
2. Check Docker Engine version >= 20.10 (minimum for buildx + proper WSL2)
   - If lower → warn, do not abort (works, but not guaranteed)
3. Check image availability: `docker images gosh-memory:{version}`
   - If registry is unreachable → abort with explicit message
```

**Platform-specific instructions when Docker is missing:**

```
macOS:
  ⟡ Docker not found. Install one of:
    • Docker Desktop: https://docs.docker.com/desktop/mac/
    • OrbStack (lightweight): https://orbstack.dev
    • Colima (CLI-only): brew install colima && colima start

Windows:
  ⟡ Docker not found. Install Docker Desktop:
    https://docs.docker.com/desktop/windows/
    Ensure WSL2 backend is enabled (Settings → General → Use WSL2).
    Windows Home: WSL2 required (Hyper-V unavailable).

Linux:
  ⟡ Docker not found.
    https://docs.docker.com/engine/install/
```

**Landlock inside Docker on macOS/Windows:**

Docker on macOS and Windows runs a Linux VM (HyperKit / WSL2). The VM kernel supports Landlock if it is recent enough (>= 5.19). `gosh-memory` applies Landlock inside the container in the standard way — the host OS does not matter.

### Download logic in CLI

```
1. Docker preflight (see above) — only for memory
2. For agent: GET GitHub API /repos/gosh-sh/gosh-ai-agent/releases/latest → manifest.json
   For memory: GET GitHub API /repos/gosh-sh/gosh-ai-memory/releases/latest → manifest.json
3. Detect the current platform's target
4. Download the archive from release assets
5. Verify SHA-256 against the manifest
6. Extract:
   - agent → /usr/local/bin/gosh-agent (sudo)
7. For memory: download Docker image tar from GitHub Release → `docker load`
```

### Auto-update

Checks for new versions on every CLI command invocation, no more than once every 12 hours.

**Mechanism:**

```
1. On startup, read ~/.gosh/agent/last_update_check (unix timestamp)
2. If less than 12 hours have passed → skip
3. If 12 hours or more have passed:
   a. Async query GitHub API releases/latest for each component (timeout: 2s, does not block the command)
   b. Compare latest version from manifest with the current one
   c. If a new version is available → print a message
   d. Write the current timestamp to last_update_check atomically (temp file + rename)
   e. On network error → silently skip, do not show an error
```

**Output:**

```
$ gosh setup
⟡ New gosh CLI available: v0.6.0 (current: v0.5.0)
⟡   curl -fsSL https://raw.githubusercontent.com/<org>/<repo>/main/install.sh | bash -s -- --version v0.6.0
...normal command output...
```

**State file:** `~/.gosh/agent/last_update_check`
Write is atomic: write temp → fsync → rename, so file corruption on crash is impossible.

**GitHub Token:** CLI automatically reads `GITHUB_TOKEN` from env for GitHub API requests (rate limit 60 → 5000 req/h).

### File Structure

**Linux / macOS:**

```
/usr/local/bin/
├── gosh                        # CLI
└── gosh-agent                  # Agent

~/.gosh/
├── memory/
│   ├── instances/              # memory instance configurations
│   │   └── {name}.toml
│   └── current                 # name of the current instance
├── agent/
│   ├── instances/              # CLI: agent instance configurations
│   │   └── {name}.toml
│   ├── current                 # CLI: name of the current agent
│   ├── state/                  # gosh-agent: per-instance runtime data
│   │   └── {name}/
│   │       ├── buffer/
│   │       │   └── pending.jsonl
│   │       └── offsets/
│   │           └── {session}.json
│   └── last_update_check       # unix timestamp of the last update check
└── run/
    ├── {service}_{name}.pid    # PID files
    └── {service}_{name}.log    # logs
```

Secrets (tokens, keys) are stored in the OS keychain via the `keyring` crate:
- macOS: Keychain
- Linux: D-Bus secret-service (GNOME Keyring / KWallet)

The agent receives secrets via `--bootstrap-file` on first launch.
The bootstrap file is deleted by the CLI after a successful health check.

**Windows:**

```
%LOCALAPPDATA%\gosh\bin\
├── gosh.exe
└── gosh-agent.exe

%LOCALAPPDATA%\gosh\
├── memory\instances\
├── agent\instances\
├── agent\current
├── agent\state\{name}\buffer\
├── agent\state\{name}\offsets\
├── agent\last_update_check
└── run\
```

Secrets are stored in Windows Credential Manager via the `keyring` crate.

**Per-project (all platforms):**

```
.gosh-memory.toml               # project configuration
.mcp.json                       # MCP server config
```

## 6. Landlock Isolation

### Requirement

Agent and memory use Landlock for self-sandboxing when available. Implementation is at the code level, not an external wrapper.

The CLI does not use Landlock — it is a short-lived user-facing process that works with arbitrary filesystem paths provided via arguments (`--data-dir`, `--bundle`, `--file`, etc.).

### Strategy

```rust
fn apply_landlock_sandbox(config: &SandboxConfig) -> Result<(), LandlockError> {
    let abi = landlock::ABI::new_current()
        .map_err(|_| { /* Landlock unavailable — continue without sandbox */ })?;

    let mut ruleset = Ruleset::default()
        .handle_access(AccessFs::from_all(abi))?
        .create()?;

    for path in &config.read_paths {
        ruleset.add_rule(PathBeneath::new(path, AccessFs::from_read(abi)))?;
    }
    for path in &config.write_paths {
        ruleset.add_rule(PathBeneath::new(path, AccessFs::from_all(abi)))?;
    }

    ruleset.restrict_self()?;
    Ok(())
}
```

### Permissions per component

**Agent (`gosh-agent`):**

```
READ:  /usr, /etc, ~/.gosh/agent/config.toml, project working directory
WRITE: ~/.gosh/agent/buffer/, ~/.gosh/agent/offsets/
NET:   LLM API, memory service, gateway
```

**Memory (`gosh-memory`, inside Docker):**

```
READ:  /usr, /etc, own data
WRITE: /var/lib/gosh-memory/
NET:   service port only (bind)
```

### Behavior when Landlock is unavailable

```
if Landlock is available:
    → "sandbox: active (Landlock ABI vN)"
else:
    → "sandbox: unavailable, running without isolation"
    → continue without restrictions
```

## 7. Memory in Docker + Landlock

```
[Host OS]
  └── Docker container (namespace isolation, cgroups)
        └── Landlock (filesystem + network restrictions)
              └── gosh-memory process
```

Docker isolates from the host, Landlock isolates inside the container — two independent layers. Minimum host kernel: 5.19+.

**macOS/Windows Docker limitation:** Docker on macOS and Windows uses virtiofs/gRPC FUSE for bind mounts between the host and the Linux VM. Landlock cannot bind rules to FUSE-backed paths, so the sandbox is automatically skipped when a write probe to the data directory fails. Docker named volumes (stored inside the VM) work correctly with Landlock. On native Linux, Landlock works with all mount types. The sandbox performs a fork-based probe at startup and only activates if the data directory is writable under Landlock.

## 8. CI/CD: Release Builds

```
Trigger: push tag v*

Jobs:
  build-{target}:           # matrix across platforms
    1. checkout
    2. setup rust + target
    3. cargo build --release --target {target}
    4. tar -czf {artifact}.tar.gz {binary}
    5. sha256sum {artifact}.tar.gz > {artifact}.tar.gz.sha256
    6. upload artifacts

  release:
    needs: [build-*]
    1. Create GitHub Release
    2. Generate manifest.json from checksums of all artifacts
    3. Attach all artifacts (.tar.gz + .sha256 + manifest.json)

  docker (memory only):
    1. docker build
    2. docker save → gosh-memory-v{version}.tar
    3. sha256sum → .sha256
    4. Generate manifest.json
    5. Attach all artifacts (.tar + .sha256 + manifest.json) to GitHub Release
```

## 9. Version Compatibility

The CLI manifest contains a `requires` field:

```json
{
  "version": "0.7.0",
  "format_version": 1,
  "requires": {
    "gosh-agent": ">=0.6.0",
    "gosh-memory": ">=0.4.0"
  },
  "artifacts": { ... }
}
```

`format_version` is a required field. CLI with an unknown format_version → graceful degradation (ignore unknown fields, use known ones).

**During `gosh setup`:**

```
1. Fetch manifest from GitHub Releases for each component
2. Compare requires with installed versions
3. If incompatible → warn BEFORE downloading:
   ⟡ gosh v0.7.0 requires gosh-agent >= v0.6.0 (installed: v0.5.0)
   → update all components? [y/N]
4. Offer to update everything at once
```

## 10. CLI Self-Update

The CLI never overwrites its own binary in place. The `gosh setup`
process **is** the gosh binary; truncating `/usr/local/bin/gosh` from
inside that process risks SIGBUS / weird crashes (Linux/macOS) or
hard refusal (Windows). The safe upgrade path runs as a *separate*
process — that's what `install.sh` does (download to tmp, atomic
`install`/rename), so the running gosh keeps its in-memory copy
until exit.

The auto-update notification therefore prints the install.sh
one-liner directly, pinned to the latest version:

```
⟡ New gosh CLI available: v0.7.0 (current: v0.6.0)
⟡   curl -fsSL https://raw.githubusercontent.com/<org>/<repo>/main/install.sh | bash -s -- --version v0.7.0
```

`gosh setup --component cli` prints the same one-liner (with
`--version` appended when set) without doing any download itself.
`gosh setup --component cli --bundle <path>` is rejected up-front —
no install.sh available offline, and we refuse to ship the unsafe
in-place overwrite.

## 11. Version Pinning / Rollback

```bash
gosh setup --version v0.5.0
```

Pins **agent and memory** to the specified version (CLI is never
installed in-place, see §10). The CLI fetches the agent/memory
manifests from `/releases/tags/v0.5.0` and downloads the
corresponding artifacts. Re-running with the same `--version` is a
no-op when the local agent binary and the local
`gosh-memory:0.5.0` Docker image both already match.

For the CLI binary itself, use the install.sh one-liner with the
target version:

```bash
curl -fsSL https://raw.githubusercontent.com/<org>/<repo>/main/install.sh | bash -s -- --version v0.5.0
```

(The branch in the `install.sh` URL — `main` above — selects which
*installer script* to fetch; `--version` selects the *binary*. They
can differ; the installer reads the manifest of the requested
release to pick the right artifact.)

## 12. Air-Gapped Installation

### Creating a bundle (on a machine with network)

```bash
# Bundle everything (CLI + agent + memory):
gosh bundle -o gosh-bundle-v0.5.0-x86_64-unknown-linux-gnu.tar.gz

# Bundle only specific components:
gosh bundle --cli --agent        # without Docker image (much smaller)
gosh bundle --cli                # CLI only
gosh bundle --memory             # memory Docker image only
```

Without flags, `gosh bundle` downloads all components. With `--cli`, `--agent`, `--memory` flags — only the specified ones.

Bundle contents:
- `bundle-meta.json` — versions, platform, which components are included
- `cli/manifest.json` + binary archive (if `--cli`)
- `agent/manifest.json` + binary archive (if `--agent`)
- `memory/manifest.json` + Docker image tars for both amd64 and arm64 (if `--memory`)

CLI and agent archives are platform-specific (bound to the host that created the bundle). Memory Docker images are bundled for both architectures so the bundle is portable — `gosh setup --bundle` selects the correct one for the target machine.

### Installing from a bundle (air-gapped machine)

```bash
# Extract CLI binary from the bundle and place in PATH:
tar xzf gosh-bundle-v0.5.0-x86_64-unknown-linux-gnu.tar.gz cli/
tar xzf cli/gosh-v0.5.0-x86_64-unknown-linux-gnu.tar.gz
sudo install gosh /usr/local/bin/gosh

# Then install agent + memory from the bundle:
gosh setup --bundle ./gosh-bundle-v0.5.0-x86_64-unknown-linux-gnu.tar.gz
```

`gosh setup --bundle` extracts the archive, verifies SHA-256 checksums from manifests, installs binaries, and loads the Docker image via `docker load`. Only components present in the bundle are installed.

## 13. Notes

**Memory on macOS/Windows:** runs via Docker (Docker Desktop, OrbStack, or Colima on macOS; Docker Desktop on Windows). `gosh setup` checks for Docker and prints platform-specific instructions if not found.

**GitHub API rate limit:** without a token — 60 req/h. CLI reads `GITHUB_TOKEN` from env automatically. For auto-update (2 requests/day) the limit is not critical.

**macOS shasum:** install.sh detects the platform and uses `shasum -a 256` instead of `sha256sum`.

**Checksum format:** GNU coreutils: `<hash>  <filename>` (two spaces).
